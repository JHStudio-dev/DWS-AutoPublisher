# Code standards

Darkward Studio standards for DWS PublishFlow. These are mandatory across the
codebase.

## TypeScript

- Strict mode on. Clear, explicit types.
- Avoid `any` unless absolutely necessary; prefer `unknown` + narrowing.
- Keep files small and responsibilities separated.
- Avoid oversized components; extract when a component does too much.
- Avoid overengineering, unnecessary abstractions, and duplicated logic.
- No hardcoded business data; no hardcoded credentials.
- Use environment variables for secrets.
- Use reusable, typed services for business logic.
- Keep database access organized and typed (generated Supabase types).

## Separation of concerns

- **UI logic** stays separate from **business logic**.
- **Auth logic** stays out of UI components (`lib/auth/`, `services/auth/`).
- **Automation logic** is completely separate from frontend components
  (`automation/`, behind interfaces).
- **Theme logic** is centralized (`lib/theme/`).
- Components depend on services; services depend on data/types; never the reverse.

## CSS naming — BEM with the `dws-` prefix (mandatory)

All project-specific classes follow BEM with the global `dws-` prefix.

```
block:           dws-dashboard
element:         dws-dashboard__header
                 dws-dashboard__content
                 dws-dashboard__card
modifier:        dws-dashboard__card--active
```

Examples by area:

```
dws-auth  dws-auth__form  dws-auth__field  dws-auth__actions  dws-auth__error
dws-vehicle-form  dws-vehicle-form__field  dws-vehicle-form__actions
dws-publication-flow  dws-publication-flow__step  dws-publication-flow__summary
dws-group-library  dws-group-library__search  dws-group-library__item
```

**Never** use generic class names such as: `container`, `card`, `wrapper`,
`section`, `button-primary`, `main-content`.

Tailwind utilities are used for layout/spacing, but semantic, named component
classes follow the `dws-` BEM convention, and **colors come from theme tokens**
(`--dws-color-*`), never hardcoded hex. See
[theme-and-branding.md](theme-and-branding.md).

## Comments

Write minimal, human-style comments. Use short section comments only when useful:

```
// Dashboard
// Auth
// Theme
// Vehicle management
// Publication workflow
// Validation
// Logs
```

Do **not** write AI-style comments such as:

```
// This function handles vehicle creation
// Generated by AI
// TODO generated automatically
// This component renders the dashboard
```

Comments explain *why* when it is not obvious — not *what* the code already says.

## UI requirements

Every feature includes:

- **Loading** states during async work and session checks.
- **Empty** states for lists with no data.
- **Error** states with user-safe messages (no raw errors leaked).
- Basic **accessibility** (labels, focus, keyboard, contrast).
- **Responsive** behavior.

Avoid: excessive animation, heavy effects, overdecorated layouts, marketing-style
visuals, unnecessary gradients, random icons, overusing red or gold.

## Validation

- Validate all user inputs with Zod schemas in `lib/validation/`.
- Validate on the server; mirror on the client for UX.
- Sanitize/escape user-generated content at render.

## Error handling

- Structured error types in `lib/errors/`.
- Log details server-side; show sanitized messages to users.
- Never surface secrets, stack traces, or raw database errors to the UI.

## Git workflow

- Commit after each meaningful milestone; keep commits focused.
- Natural, human commit messages, e.g.:

```
feat: add auth architecture
feat: create dashboard structure
feat: add vehicle management module
fix: protect dashboard routes
refactor: separate auth services
chore: organize project structure
```

- Do **not** create one huge commit, mix unrelated work, commit secrets, or
  commit `.env` files.

## Quality control before closing a phase

Check: TypeScript safety, database consistency, Supabase usage, auth flow,
protected routes, RLS assumptions, naming consistency, BEM class naming, theme
consistency, component size, service separation, error handling, form validation,
empty/loading/error states, responsive behavior, security risks, and future
SaaS / public-catalog compatibility.

Run the available scripts before committing:

- `npm run lint`
- `npm run typecheck`
- `npm run build`
- `npm run test` (when present)

If a script does not exist yet, document the gap and propose adding it in the
phase that needs it.
